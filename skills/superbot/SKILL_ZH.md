# SuperBot — 产品驱动的 AI 编程协作技能

## 触发方式

### 手动触发
用户在对话中输入 `/superbot` 调用本技能。所有支持技能的 Agentic 工具均使用此方式。

### 配置命令
用户输入 `/superbot config` 可交互式更新配置文件。

### 初始化命令
用户输入 `/superbot init` 强制执行项目配置初始化。适用场景：
- 第一次在项目中使用 SuperBot
- 需要从头重新初始化配置

**执行逻辑：**
- 如果 `.superbot/superbot.json` 已存在：展示当前配置并询问用户是否要重新初始化
- 如果配置不存在：引导用户完成交互式初始化（与首次使用流程相同）
- 初始化完成后，检查并创建缺失的 SPEC 文件（使用模板）

### 更新 Spec 命令
用户输入 `/superbot update spec` 触发基于当前 session 对话历史的 SPEC 文件批量提取与更新。

**执行流程：**
1. 收集当前 session 中用户和 AI 的所有对话
2. 提取其中有价值的产品决策（需求、架构、UI 规范等）—— 面向产品终态需要被沉淀的内容
3. 提炼为结构化内容，更新对应的 SPEC 文件
4. 通知用户更新了哪些内容（如："已更新 01-产品定义.md：新增核心用户群定义；已更新 03-功能清单.md：新增 2 个功能点"）

### 自动触发（按平台）
不同 Agentic 工具在新 session 启动时自动触发本技能的配置方式：

| 平台 | 自动触发方式 |
|------|-------------|
| **Claude Code** | 在 `~/.claude/settings.json` 中配置 `SessionStart` 钩子（matcher: `startup`），用 `cat` 直接将 SKILL.md 内容注入 Claude 上下文 |
| **OpenClaw** | 在 OpenClaw 的配置文件中设置 session 启动钩子，指向本技能路径 |
| **OpenCode** | 在 OpenCode 的配置中设置 session init hook，加载本技能 |
| **Qoder** | 在 `.qoder/skills/` 目录下安装技能后，通过 session 启动配置自动加载 |
| **其他平台** | 使用平台提供的 session 生命周期钩子，在 session 创建时加载本 SKILL.md |

> 如果自动触发未配置，技能仍可通过 `/superbot` 手动调用。

### 帮助命令
用户输入 `/superbot help` 或 `/superbot h` 时，输出所有可用命令及对应能力的快速参考。

**当用户触发时，直接输出：**

```
SuperBot 可用命令：

/superbot              触发技能，开始产品驱动的协作流程
/superbot h            显示本帮助
/superbot config       交互式更新项目配置（spec_dir 等）
/superbot init         强制初始化配置和缺失的 SPEC 文件
/superbot update spec  从当前 session 对话中批量提炼产品决策，写入 SPEC

SPEC 文件体系（位于 ${spec_dir}）：
01-产品定义.md     产品定位、目标、用户群
02-用户故事.md     用户故事、角色、场景
03-功能清单.md     功能模块、优先级、依赖
04-UI设计规范.md   交互流程、页面结构、UI 规范
05-技术架构规范.md  技术选型、架构、非功能性要求
06-其他.md         其他产品认知

使用方式：随时在对话中做决策，我会自动将确认的决策实时写入对应 SPEC 文件。
```

## 一、协作模式与权限

### 1. 激活确认

会话开始时，**必须**明确告知用户本技能已被激活，例如：

> SuperBot 技能已激活。

### 2. using-superpowers 协作

SuperBot 可独立工作。如果安装了 `using-superpowers` 技能，会自动协同以获得更好的体验。

**不要在 session 启动 hook 阶段尝试激活 `using-superpowers`。** 而是在**用户发送第一条消息时**再尝试激活。如果激活成功，在后续交互中优先通过该技能处理请求。

如果 `using-superpowers` 技能未安装或无法激活，**不要阻断流程**。直接继续正常执行，无需警告。

### 3. 文件读取权限

对本项目根目录及子目录中所有文件的 `读取` 操作，不需要经过用户允许。这是基本工作权限。

### 4. 配置管理

#### 配置文件位置

读取当前项目目录下的 `.superbot/superbot.json` 文件。

| 字段 | 说明 | 默认值 |
|------|------|--------|
| `spec_dir` | SPEC 目录，用于沉淀产品核心定义 | `./spec` |

#### 配置初始化

**如果 `.superbot/superbot.json` 已存在**：静默读取并使用配置。不要提示用户确认。

**如果 `.superbot/superbot.json` 不存在**：引导用户完成初始化：
1. 说明 SuperBot 的作用和每个配置字段的用途
2. **逐个**询问用户确认或自定义每个字段
3. 所有字段确认后，创建配置文件
4. 继续执行后续启动流程

#### 交互式配置更新（`/superbot config`）

当用户调用 `/superbot config` 时：
1. 读取当前配置（如文件不存在则使用默认值）
2. 展示每个字段的当前值
3. **逐个**询问用户确认或修改每个字段
4. 保存更新后的配置到 `.superbot/superbot.json`
5. 确认更新完成

## 二、SPEC 文件体系

### 1. 文件清单

在 `${spec_dir}` 目录下维护以下 6 个文件（缺失则从模板创建）：

| 文件 | 用途 |
|------|------|
| `01-产品定义.md` | 产品定位、整体目标、核心用户群 |
| `02-用户故事.md` | 用户故事、角色、使用场景 |
| `03-功能清单.md` | 功能模块划分、优先级、依赖关系 |
| `04-UI设计规范.md` | 交互流程、页面结构、UI 规范、组件约束 |
| `05-技术架构规范.md` | 技术选型、架构风格、中间件、非功能性要求 |
| `06-其他.md` | 其他对产品认知有帮助的信息 |

模板文件位于 `skills/superbot/templates/zh/`。创建新的 SPEC 文件时，使用这些模板作为初始内容。

### 2. Session 启动流程

**重要：SPEC 目录是用户最容易搞错的地方。每次 session 开始，必须显式告知用户 SPEC 目录的完整路径和文件状态。**

> 注意：以下流程（读取 SPEC、沟通冲突）在用户发送第一条消息后执行。如果当前处于 session 启动 hook 阶段且无用户交互上下文，仅完成文件读取，将所有冲突/疑问暂存，等用户消息到来时再统一提出。

在 **每一次新 session 开始时**，执行以下动作：

1. **读取并分析** `${spec_dir}` 内的所有 SPEC 文件；
2. **输出启动摘要**（在用户发送第一条消息后，第一时间输出）：
   - 告知用户 SPEC 目录的完整路径（例如：`SPEC 目录：./spec`）；
   - 列出已存在的 SPEC 文件和缺失的文件；
   - 如果有缺失文件，告知用户将从模板自动创建；
3. **构建统一理解**，包括：
   - 产品定位与整体目标；
   - 核心用户群与关键用户故事；
   - 功能模块划分及优先级；
   - UI/交互的约束与风格；
   - 技术架构、技术栈、非功能性要求；
4. **主动沟通**：若发现以下情况，必须向用户提出并确认：
   - 内容不理解、存在多种解释可能；
   - 同一文档内部存在自相矛盾；
   - 不同文档之间对同一概念/功能/流程定义冲突；
   - SPEC 与当前代码实现明显不一致；

   沟通时说明冲突点的具体位置，给出 1–2 种可能解释，请用户选择或补充。

**只有在对 SPEC 形成清晰、一致的理解后，才开始具体设计与编码建议。**

## 三、SPEC 实时沉淀规范

SPEC 目录是需要 **持续迭代维护** 的产品知识库。目标：
> 让任何后来加入的成员或 Agent，只需阅读这套 SPEC，就能快速对齐共识并复制出产品终态。

### 实时沉淀原则

**在每一轮对话中，必须实时识别、提炼、写入关键决策到 SPEC 文件。**

流程：

1. **监听**：在用户回复中识别新的决策、确认、变更或约束；
2. **判断**：按下方"沉淀判定矩阵"判断是否值得写入；
3. **提炼**：将口语化讨论抽象为结构化结论；
4. **写入**：直接 Edit/Write 对应 SPEC 文件；
5. **告知**：在回复中简短说明"已更新 XX.md：新增了 YY 条目"。

#### 沉淀判定矩阵

| 必须写入 | 不写入 |
|---------|--------|
| 用户明确确认的产品决策（"就这么定""对""是的"） | 讨论过程中的试探性想法（"要不试试""如果……呢"） |
| 用户新增/修改/删除的功能需求 | 技术实现细节的试错（除非影响架构决策） |
| 用户确认的用户故事、角色、场景变化 | 纯调试、排错、代码修复过程 |
| 用户确认的技术选型或架构决策 | 临时的、明确标注为"先这样后面再改"的权宜方案 |
| 用户确认的 UI/交互规范 | 已被用户明确否决的方案 |
| 用户提出的非功能性约束（性能、安全、合规等） | 纯闲聊、与产品无关的技术讨论 |

**核心判断标准**：这条信息是否会影响后来者对产品"应该做成什么样"的理解？如果是，写入。

#### 写入时机

- **即时写入**：用户给出明确决策/确认时，直接完成 SPEC 更新
- **批量写入**：一轮对话包含多个相关决策时，合并为一次更新
- **会话结束前回顾**：对话接近尾声时，主动检查是否有遗漏决策需要补写

### 更新内容要求

1. 直接 Edit/Write 对应 SPEC 文件；
2. 给出结构化的更新内容（新增章节/小节标题、功能条目等）；
3. 用终态视角撰写，避免：
   - 大量口语化描述；
   - 与时间或对话上下文强耦合的表达；
   - 含糊或容易产生歧义的措辞。

### 处理 SPEC 与实现的偏差

发现以下情况时：
- 实际代码实现与 SPEC 不一致；
- 新需求与当前 SPEC 冲突；
- SPEC 已过时或不再合理；

应当：
1. 明确标记冲突点；
2. 向用户询问：以新对话结论为准还是以 SPEC 为准；
3. 用户决策后，相应更新 SPEC 或调整实现建议。

## 四、综合工作流

1. **帮助（`/superbot help`）**：
   - 展示所有可用命令及对应能力；
   - 列出 SPEC 文件清单与简要说明；
   - 说明使用方式。

2. **启动 / 新 session**：
   - 如果 `using-superpowers` 可用，激活它；
   - 读取 `.superbot/superbot.json`：
     - 如果存在：静默使用
     - 如果不存在：引导用户完成交互式初始化
   - 检查并创建缺失的 SPEC 文件（使用模板）；
   - 读取并分析所有 SPEC 文件；
   - 如有冲突或疑问，先沟通确认。

3. **配置更新（`/superbot config`）**：
   - 展示当前配置；
   - 引导用户逐个更新每个字段；
   - 保存更新后的配置。

4. **强制初始化（`/superbot init`）**：
   - 如果配置已存在：展示当前配置并询问用户是否要重新初始化；
   - 如果配置不存在：引导用户完成交互式初始化（与首次使用流程相同）；
   - 初始化完成后，检查并创建缺失的 SPEC 文件（使用模板）；
   - 向用户确认完成。

5. **批量更新 SPEC（`/superbot update spec`）**：
   - 收集当前 session 中用户和 AI 的所有对话；
   - 提取其中有价值的产品决策（面向产品终态需要被沉淀的内容）；
   - 按照”沉淀判定矩阵”提炼为结构化内容；
   - 更新对应的 SPEC 文件；
   - 通知用户更新摘要（哪些文件、什么内容）。

6. **设计与开发**：
   - 所有方案、接口、页面、架构建议与 SPEC 保持一致；
   - 如需偏离或扩展 SPEC，必须指出并征求用户确认。

7. **知识库维护**：
   - 在对话中持续抽取可沉淀的产品认知；
   - 直接写入对应 SPEC 文档；
   - 通过不断迭代 SPEC，保持统一认知。

以上规则在整个项目生命周期内持续生效。
